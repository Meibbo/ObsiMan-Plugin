ObsiMan — Brainstorm Scope

14 sistemas identificados. Los agrupé por versión objetivo para que empecemos por lo más urgente.

¿Por dónde empezamos el brainstorm?

Pediste context menus primero — confirma el alcance antes de la primera pregunta.

A

Context Menus — solo in-frame

Unificar y mejorar los menús dentro de ObsiMan (Tags/Props/Files panels). Crear un ContextMenuService interno con API limpia.

Pros

  • Alcance acotado
  • No toca el workspace nativo
  • Bloquea menos features

Cons

  • No da entry point desde notas
B

Context Menus — solo workspace nativo

"Abrir en ObsiMan", "Enviar a filtro", "Ver prop en explorer" desde el right-click nativo de Obsidian (file-menu, editor-menu, more-options).

Pros

  • Entry point poderoso para usuarios
  • Muy visible — primero que ven

Cons

  • Depende de APIs internas para visibility control
C

Context Menus — arquitectura unificada (recomendado)

Un ContextMenuService que sirve a ambos: registra acciones con metadatos (label, icon, cuando aplica, handler), y las distribuye tanto a los menús in-frame como a los workspace menus de Obsidian. Una API, dos superficies.

Pros

  • No duplicas código al agregar acciones nuevas
  • Context menu visibility (punto 11) se implementa aquí
  • Escala con el plugin

Cons

  • Más diseño inicial
  • Una sesión más larga
