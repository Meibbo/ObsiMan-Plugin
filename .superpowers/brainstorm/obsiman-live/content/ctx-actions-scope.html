ContextMenuService — Tipos de acciones

Primera pregunta: ¿qué tipo de objetos pueden recibir acciones de contexto?

El servicio necesita saber sobre qué registras una acción para saber cuándo mostrarla. Tres modelos posibles:

A

Solo por tipo de nodo (node-centric)

Registras acciones contra tipos: tag, prop, value, file, folder. El servicio muestra las acciones registradas para ese tipo — en el panel in-frame Y en el workspace nativo cuando Obsidian expone ese objeto.

Pro

  • Simple de entender
  • Consistente in-frame / workspace

Con

  • Menos granularidad (no distingue contexto)
B

Por tipo de nodo + superficie (node + surface)

Registras acciones con dos dimensiones: el tipo de objeto (file, tag…) y la superficie donde aparece (panel, file-menu, editor-menu). Una acción puede estar en ambas o solo en una.

Pro

  • Máxima flexibilidad
  • Context menu visibility entra naturalmente

Con

  • Más API para aprender
C

Empezar con A, expandir a B cuando haga falta RECOMENDADO

v1.0.0 usa el modelo node-centric (simple). La estructura interna ya contempla la dimensión de superficie para que agregar visibilidad/control en v1.1+ no requiera reescribir nada.

Contexto actual del codebase

Hoy, cada Panel implementa sus propios menús inline:
TagsExplorerPanel → rename/delete/send-to-frontmatter
PropsExplorerPanel → rename prop / add value / convert type
FilesExplorerPanel → rename / delete / move

El servicio los centralizaría y añadiría las mismas acciones al workspace nativo.
